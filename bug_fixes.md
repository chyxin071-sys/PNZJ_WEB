# CM1.0 Bug 修复记录与版本计划

## 待修复 Bug 列表 (待确认)

以下是本次整理的 13 个优化与修复需求，将按照您确认后的顺序逐一修复：

### 1. [x] 弹窗导航栏遮挡问题
**问题描述**：新建线索、新建待办的时候，底部的自定义导航栏没有隐藏，导致遮挡或体验不佳。用户期望它像背景一样被黑色半透明遮罩覆盖，而不是直接消失导致画面跳动。
**解决方案**：引入微信小程序原生的 `<root-portal>` 组件。将所有涉及全屏遮罩的弹窗（新建待办、高级筛选、新建线索、新建工地、恭喜开单等）包裹在 `<root-portal>` 内部。这样弹窗会被渲染到页面的最高层级，自然地将下方的 `custom-tab-bar` 和内容一起覆盖在半透明黑色遮罩下方，无需调用 `setData` 强行控制隐藏/显示。
**状态**：✅ 已修复（重构了 `pages/leads/index`、`pages/index/index`、`pages/projects/index` 和 `pages/leadDetail/index` 中弹窗逻辑并移除了旧的隐藏代码）。

### 2. [x] 新建工地交互统一
**问题描述**：新建工地目前不是弹窗/浮窗形式，与新建待办和新建客户不统一。
**解决方案**：将新建工地（或从线索转化工地）的页面交互重构为底部升起的浮窗组件，保持全系统的新建操作体验一致。
**状态**：✅ 已修复（重构了 `pages/projects/index.wxml` 为底部弹窗，删除了旧的 `pages/projectForm`，并在 `leadDetail` 中增加了自动弹窗跳转逻辑）。

### 3. [x] 待办列表排序逻辑
**问题描述**：待办排序没有按照最新时间在最上面。
**解决方案**：修改待办列表获取数据（`pages/index/index.js`）的排序逻辑，强制加入 `orderBy('createdAt', 'desc')` 或 `updatedAt` 倒序，确保最新添加的待办始终置顶。
**状态**：✅ 已修复（在 `pages/index/index.js` 的 `fetchData` 中添加了 `.orderBy('createdAt', 'desc').limit(100)`）。

### 4. [x] 跟进记录编辑问题
**问题描述**：编辑跟进内容时，之前的内容没显示，且编辑保存不成功；未显示“已编辑”标签。
**解决方案**：修复 `pages/addFollowUp/index` 的回显逻辑，确保进入页面时能正确拉取并显示旧内容；修复保存逻辑使用 `update` 且记录 `editedAt` 时间戳；在UI层面，只要存在 `editedAt`，就在时间旁边渲染“已编辑”的小标签。
**状态**：✅ 已修复（在 `pages/leadDetail/index.wxml` 中修复了 `editFollowUp` 绑定的 `data-id` 参数，解决了因 id 丢失导致的回显和更新失败问题）。

### 5. [x] 设计排期按钮显示逻辑
**问题描述**：还未开启设计工作流时，客户详情页底部不应该显示“编辑排期”的按钮。
**解决方案**：在 `leadDetail/index.wxml` 中增加条件判断，只有当 `lead.designNodes` 存在且长度大于 0（即已开启设计工作流）时，才渲染底部的“编辑排期”按钮。
**状态**：✅ 已修复（在 `leadDetail/index.wxml` 底部操作栏中增加了 `&& lead.designNodes && lead.designNodes.length > 0` 的判断）。

### 6. [x] 微信订阅通知缺失
**问题描述**：添加跟进时没有触发微信订阅通知，很多其他操作也没有。
**解决方案**：全面排查订阅消息的触发点（如添加跟进、节点验收、状态变更），确保每次操作前正确调用了 `requestSubscribe()` 收集权限，并在云端/本地正确调用微信接口发送模板消息。
**状态**：✅ 已修复（已在跟进记录、节点验收、客户分享等关键操作中接入 `requestSubscribe` 和 `sendSubscribeMessage`）。

### 7. [x] 未读跟进记录红点提示
**问题描述**：未读的跟进记录没有红点提示。
**解决方案**：引入未读消息检查机制。在客户列表、详情页入口或者消息通知页，根据 `notifications` 集合中当前用户的 `isRead: false` 数据，在对应的 UI 位置（如 TabBar 或卡片右上角）渲染红色小圆点。
**状态**：✅ 已修复（客户列表和详情页“跟进记录”Tab上已增加红点提示，点击跟进记录Tab时自动标记为已读）。

### 8. [x] 微信消息截断问题
**问题描述**：微信消息显示的内容到一半就停止了（如“什么什么工序已...”）。
**解决方案**：排查发送订阅消息或系统通知时对 `content` 的截取逻辑（目前代码里有 `substring(0, 30)` 或 `substring(0, 40)` 的限制），放宽截取长度或优化文案结构，确保核心信息能在微信通知卡片内完整展示。
**状态**：✅ 已修复（优化了工序验收的订阅消息文案结构，提取核心工序名称，避免被前缀地址等无用信息占据20字符限制导致截断）。

### 9. [x] 无需签字工序的提示优化
**问题描述**：不需要签字的普通工序，验收完不要再弹出“待客户确认”的提示了。
**解决方案**：在 `projectDetail/index.js` 的 `submitAcceptance` 方法中修改判断，如果是普通工序（非首尾节点），提交后直接 Toast 提示“已完成/已提交”，不再显示“待客户确认”。
**状态**：✅ 已修复（优化了 `submitAcceptance` 成功后的提示语，仅当 `status` 为 `awaiting_signature` 时才提示“待客户确认”，否则提示“已验收通过”）。

### 10. [x] 图片加载极慢问题
**问题描述**：图片加载很慢，员工端和客户端都慢，客户端甚至加载不出来。
**解决方案**：
1. 上传端：彻底落实 `wx.compressImage`（目前可能有些地方遗漏或参数不对），强制所有图片在上传前压缩。
2. 渲染端：将图片 URL 替换为腾讯云存储的图片处理样式参数（如在 url 后追加 `?imageMogr2/thumbnail/600x/interlace/1`），强制输出小体积的 WebP 或缩略图，极大提升加载速度。
**状态**：✅ 已修复（在 `utils/format.wxs` 中增加了图片 URL 处理逻辑，在 `projectDetail` 和 `projectShare` 中均采用了该格式化方式，同时补充了文件上传前的图片压缩流程）。

### 11. [x] 客户签字后的反向通知
**问题描述**：客户在分享页签完字提交后，员工端（项目经理、销售等）没有收到微信和系统的通知。
**解决方案**：在客户端签字提交成功的云函数或逻辑中，新增通知分发代码。签字成功后，向该工地的相关负责人（manager, sales, designer, admin）发送“客户已签字确认”的系统通知和微信订阅消息。
**状态**：✅ 已修复（在 `pages/projectShare/index.js` 的 `sendSignatureNotifications` 中已实现对相关负责人的 `followUps` 系统记录写入、`notifications` 内部通知分发以及 `sendSubscribeMessage` 微信订阅消息推送）。

### 12. [x] 工序自动开始逻辑错误
**问题描述**：开工之后后面所有的工序的实际开工时间都自动确认了，应该是完成一项开一项。
**解决方案**：修复 `recalculateGantt`（重新计算甘特图）算法。排期算法应该只负责推算“计划时间（startDate/endDate）”，绝对不能自动把未来所有工序的 `actualStartDate`（实际开工时间）填上，也不能把未来的工序状态提前改为 `current`。只有上一项 completed 时，下一项才变 current 并记录 actualStartDate。
**状态**：✅ 已修复（优化了 `confirmStartProject` 及 `projects/index.js` 的排期生成算法，仅自动填充当前第一个工序的 `actualStartDate` 并设置为 `current` 状态，其余工序保持 `pending`，严格遵守“完成一项开一项”的顺延逻辑）。

### 13. [x] 无需签字工序的分享卡片错误
**问题描述**：不需要签字的工序，分享卡片显示成了大 LOGO，而不是预期的“已完工+项目经理+完工时间”。
**解决方案**：修复 `pages/projectDetail/index.js` 中的 `generateShareImage` Canvas 绘制逻辑。目前它可能因为状态判断遗漏（比如以为只有待确认才画字），导致普通工序走入了 fallback（直接给大 LOGO）。需确保明确判断 `status === 'completed'` 且无需签字时，正确绘制对应的文本和排版。
**状态**：✅ 已修复（在 `submitAcceptance` 验收完成后，当自动弹出工序详情弹窗时，补充调用了 `this.generateShareImage(sub)`。由于之前缺少这行代码，导致刚验收完立刻点击分享时，分享卡片未能及时生成而降级显示了公司大 LOGO）。
