# 品诺筑家 (PNZJ) 系统演进与架构复盘 (Product Evolution & Retrospective)

> **文档定位**：不同于 PRD（需求蓝图）和 Changelog（代码流水账），本文档专门用于记录 PNZJ 系统从 0 到 1 过程中的**重大战略决策、跨端架构演进（Web 与小程序的重心切换）以及背后的产品思考**。

---

## 阶段一：Web 端基建期 —— 搭建复杂的“底层大本营”
* **时间节点**：项目启动初期
* **核心目标**：解决 20 人装修团队极度依赖 Excel、数据不互通、管理混乱的问题。
* **业务动作**：
  - 采用 `Next.js 14` 搭建了 Web 网页端。
  - 在云数据库中构建了最核心的业务模型：线索状态机（A/B/C/D）、报价单核算逻辑、工地 8 大节点排期、以及多角色权限（老板/销售/设计/工长）。
* **产品反思**：Web 端屏幕大，非常适合老板看数据看板、财务核算报价单。在这个阶段，我们把整套系统的“地基”打得非常扎实。

## 阶段二：小程序突围期 —— 解决“最后一公里”的现场痛点
* **时间节点**：核心业务逻辑跑通后
* **核心痛点**：系统推向一线时遭遇阻力。工长在满是灰尘的工地不方便用电脑/网页打卡；客户每次查看工地都需要点开外部链接，体验割裂且缺乏微信强提醒。
* **战略转向**：**全面向微信原生小程序倾斜**。
* **业务动作**：
  - 开发了工长专属的“极简打卡”界面，打开微信就能传现场照片。
  - 引入微信原生能力：`getPhoneNumber` 实现业主免密静默授权查看工地。
  - 接入微信订阅消息：取代无效的微信群，将“已签单”、“工序验收”等关键节点直接推送到相关人员的微信服务通知。
* **产品反思**：这是一个极具价值的业务转向。管理系统不能只方便老板，必须让一线干活的人（工长）觉得“傻瓜、好用”，系统才能真正落地。

## 阶段三：深水区重构与双端协同 —— 贴合真实的复杂业务
* **时间节点**：系统深度使用期（近期高频迭代）
* **核心痛点**：真实的业务场景比初期设想的更复杂（如：工序不是单行道而是并行的、主材价格不能让客户看到实时变动、公司增加了财务角色）。
* **业务动作**：
  - **大节点流转重构**：打破线性排期，设计“首尾特殊化工序”，实现水电泥木油等子工序的**并行激活**与实际完工后的**动态顺延**。
  - **主材快照解耦**：首创“快照(Snapshot)分享”机制，客户看到的确认单与员工后台的数据完全解耦，保护公司价格隐私。
  - **多角色与财务脱敏**：重构分配机制，支持多名销售/设计师共同负责；新增财务角色，对未签单客户信息全站打码。
* **产品反思**：这个阶段的更新既有“新增功能”，也有对旧架构的“推翻重构”。所有的修改都是为了让系统无限逼近“真实的工地流转规律”。

## 阶段四：未来的收网与大盘管理 —— Web 端的回填
* **当前状态**：Web 端的合同模块仍为假数据，部分新增字段（如入户密码）在 Web 端尚未展示，密码安全（明文存储）仍需优化。
* **未来规划**：
  - 在小程序端彻底打通“线索 -> 报价 -> 施工 -> 交付”闭环后，重心将再次回归 Web 端。
  - 将 Web 端打造为强大的“PC 控制台”，补齐分页、复杂表格导出、全局数据看板与财务核算等重度管理功能。
