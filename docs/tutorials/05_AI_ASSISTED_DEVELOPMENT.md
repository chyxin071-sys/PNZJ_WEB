# AI辅助开发指南

> 本指南详细讲解如何使用AI（如Claude）辅助开发微信小程序，提高开发效率，减少错误。

---

## 第一部分：AI辅助开发概述

### 1.1 为什么用AI开发

**优势**：

**1. 提高开发效率**
- 快速生成代码框架
- 自动补全重复代码
- 快速实现常见功能
- 节省大量时间

**2. 降低学习成本**
- 不需要精通所有技术
- AI可以解释复杂概念
- 边做边学
- 快速上手新技术

**3. 减少错误**
- AI可以发现潜在问题
- 提供最佳实践建议
- 自动修复常见错误
- 提高代码质量

**4. 解决问题**
- 快速定位Bug
- 提供解决方案
- 解释错误原因
- 避免重复踩坑

**劣势**：

**1. 可能生成错误代码**
- AI不是万能的
- 需要人工检查
- 可能不符合需求
- 需要调试

**2. 依赖性**
- 过度依赖AI
- 不利于深入学习
- 可能养成坏习惯

**3. 理解偏差**
- AI可能理解错需求
- 需要明确描述
- 可能需要多次沟通

### 1.2 AI能做什么

**✅ AI擅长的事情**：

1. **生成代码框架**
   - 页面结构
   - 组件模板
   - 数据库设计
   - API接口

2. **实现常见功能**
   - 列表页面
   - 表单提交
   - 数据查询
   - 文件上传

3. **修复Bug**
   - 语法错误
   - 逻辑错误
   - 性能问题
   - 兼容性问题

4. **优化代码**
   - 重构代码
   - 提高性能
   - 改进可读性
   - 统一风格

5. **解释代码**
   - 解释复杂逻辑
   - 说明原理
   - 提供文档
   - 回答问题

### 1.3 AI不能做什么

**❌ AI不擅长的事情**：

1. **理解复杂业务逻辑**
   - 需要人工设计
   - AI只能辅助实现

2. **做产品决策**
   - 功能优先级
   - 用户体验设计
   - 商业决策

3. **完全替代测试**
   - 需要人工测试
   - 验证功能正确性
   - 发现边界问题

4. **保证代码100%正确**
   - 需要人工审查
   - 可能有隐藏问题

### 1.4 如何选择AI工具

**推荐工具**：

**1. Claude（推荐）**
- 优点：代码能力强、理解能力好、上下文长
- 缺点：需要付费
- 适合：复杂项目、深度开发

**2. ChatGPT**
- 优点：通用性强、知识面广
- 缺点：代码能力略弱于Claude
- 适合：一般项目、学习使用

**3. GitHub Copilot**
- 优点：IDE集成、实时补全
- 缺点：只能补全，不能对话
- 适合：日常编码

**4. Cursor**
- 优点：IDE + AI结合
- 缺点：需要付费
- 适合：全栈开发

## 第三部分：如何调试AI代码

### 3.1 理解AI生成的代码

#### 步骤1：通读代码

```javascript
// AI生成的代码
Page({
  data: {
    users: [],
    loading: false
  },

  onLoad() {
    this.loadUsers()
  },

  async loadUsers() {
    this.setData({ loading: true })
    try {
      const db = wx.cloud.database()
      const res = await db.collection('users').get()
      this.setData({ users: res.data })
    } catch (err) {
      console.error(err)
      wx.showToast({ title: '加载失败', icon: 'none' })
    } finally {
      this.setData({ loading: false })
    }
  }
})
```

**理解要点**：
1. 页面加载时调用 loadUsers
2. 显示加载状态
3. 从数据库获取用户
4. 错误处理
5. 隐藏加载状态

#### 步骤2：检查关键部分

```javascript
// 检查数据库查询
const res = await db.collection('users').get()
// ✅ 集合名称正确吗？
// ✅ 查询条件正确吗？
// ✅ 有没有分页？

// 检查数据更新
this.setData({ users: res.data })
// ✅ 字段名称正确吗？
// ✅ 数据格式正确吗？

// 检查错误处理
catch (err) {
  console.error(err)
  wx.showToast({ title: '加载失败', icon: 'none' })
}
// ✅ 错误提示友好吗？
// ✅ 有没有记录日志？
```

#### 步骤3：测试代码

```javascript
// 1. 正常情况测试
// - 数据库有数据
// - 网络正常
// - 预期：显示用户列表

// 2. 异常情况测试
// - 数据库为空
// - 网络断开
// - 数据格式错误
// - 预期：显示友好提示
```

### 3.2 发现问题

#### 常见问题类型

**1. 语法错误**

```javascript
// ❌ AI生成的代码
const users = await db.collection('users').get(
// 缺少右括号

// ✅ 修复
const users = await db.collection('users').get()
```

**2. 逻辑错误**

```javascript
// ❌ AI生成的代码
if (user.age > 18) {
  console.log('未成年')  // 逻辑反了
}

// ✅ 修复
if (user.age >= 18) {
  console.log('成年')
}
```

**3. 性能问题**

```javascript
// ❌ AI生成的代码
for (let i = 0; i < 1000; i++) {
  this.setData({ count: i })  // 频繁调用
}

// ✅ 修复
let count = 0
for (let i = 0; i < 1000; i++) {
  count = i
}
this.setData({ count })
```

**4. 不符合需求**

```javascript
// ❌ AI生成的代码
// 需求：按创建时间倒序
data.sort((a, b) => a.createdAt - b.createdAt)  // 升序

// ✅ 修复
data.sort((a, b) => b.createdAt - a.createdAt)  // 倒序
```

### 3.3 修复问题

#### 方法1：告诉AI具体问题

```
"你生成的代码有问题：
1. 第10行缺少右括号
2. 第20行逻辑反了，应该是 >= 18
3. 第30行性能有问题，频繁调用 setData

请修复这些问题"
```

#### 方法2：提供错误日志

```
"代码运行报错：
TypeError: Cannot read property 'name' of undefined
at pages/users/index.js:50

这是第50行的代码：
const name = user.name

请帮我修复"
```

#### 方法3：自己修复并让AI检查

```
"我修复了代码，请帮我检查是否正确：

修复前：
if (user.age > 18) {
  console.log('未成年')
}

修复后：
if (user.age >= 18) {
  console.log('成年')
}

这样修复对吗？"
```

### 3.4 优化代码

#### 让AI优化

```
"这段代码可以优化吗？

当前代码：
for (let i = 0; i < users.length; i++) {
  if (users[i].age > 18) {
    adults.push(users[i])
  }
}

优化目标：
1. 提高可读性
2. 使用现代JavaScript语法"
```

#### AI可能的优化

```javascript
// 优化后的代码
const adults = users.filter(user => user.age > 18)
```

## 第五部分：推荐的 AI 工具和技能（Skills）

### 5.1 什么是 Skills？

Skills 是 Claude 的扩展能力，就像给 AI 安装"插件"，让它在特定场景下更专业。

**为什么需要 Skills？**
- 让 AI 在特定任务上更专业
- 提供标准化的工作流程
- 减少重复性的提示词
- 提高开发效率

### 5.2 推荐的 Skills

#### Skill 1：brainstorming（头脑风暴）

**功能**：在写代码前，帮你理清需求、设计方案

**使用场景**：
- 要做新功能，但不知道怎么设计
- 需要技术选型建议
- 想评估方案的优缺点
- 不确定如何拆解任务

**使用方法**：
```
/brainstorming

或者直接说：
"我想做一个客户管理功能，帮我设计一下"
```

**优点**：
- 避免直接写代码导致返工
- 提供多个方案对比
- 考虑边界情况和潜在问题
- 帮助理清思路

**示例对话**：
```
你：我想做一个工地进度分享功能，业主可以查看工地照片和进度

AI：让我帮你梳理一下需求...

1. 访问方式
   - 业主如何访问？（二维码/链接）
   - 需要登录吗？

2. 身份验证
   - 如何验证是业主本人？（手机号验证）
   - 非业主能看吗？（需要申请）

3. 可见内容
   - 可以看到哪些信息？（照片/进度）
   - 不能看到什么？（报价/成本）

4. 权限管理
   - 员工如何管理权限？（审批机制）
   - 权限有效期多久？

5. 技术方案
   - 使用小程序分享页
   - 云数据库存储申请记录
   - 订阅消息通知审批结果
```

#### Skill 2：systematic-debugging（系统化调试）

**功能**：遇到 Bug 时，系统化地排查问题

**使用场景**：
- 代码报错，不知道原因
- 功能不符合预期
- 数据显示异常
- 性能问题

**使用方法**：
```
/systematic-debugging

或者直接说：
"跟进记录排序乱了，帮我查一下"
```

**优点**：
- 不会盲目猜测，而是系统化排查
- 检查日志、数据、代码逻辑
- 给出根本原因和解决方案
- 避免治标不治本

**示例对话**：
```
你：小程序端跟进记录排序乱了

AI：让我系统化排查这个问题...

步骤1：检查数据源
- 查看数据库中的 createdAt 字段
- 发现有三种格式：对象、数字、字符串

步骤2：检查排序代码
- 找到 leadDetail/index.js 第80行
- 发现用 String() 转换后比较
- 问题：对象转字符串变成 "[object Object]"

步骤3：根本原因
- 时间格式不统一导致排序失效

步骤4：解决方案
- 创建统一的时间解析函数
- 兼容三种格式
- 统一转为时间戳后排序
```

#### Skill 3：verification-before-completion（完成前验证）

**功能**：在声称"完成"之前，强制验证功能是否真的可用

**使用场景**：
- 修复 Bug 后，确认真的修好了
- 开发新功能后，确认能正常运行
- 避免"看起来对，实际不行"的情况

**优点**：
- 减少返工
- 确保代码可以直接运行
- 发现潜在问题
- 提高代码质量

**工作流程**：
```
1. AI 完成代码修改
   ↓
2. 自动触发 verification-before-completion
   ↓
3. 检查代码语法
   ↓
4. 模拟数据测试
   ↓
5. 确认修复成功
   ↓
6. 报告给用户
```

#### Skill 4：planning-with-files（文件化计划）

**功能**：把复杂任务拆解成多个步骤，记录在文件中，逐步执行

**使用场景**：
- 大型功能开发（如"做一个完整的客户管理模块"）
- 多文件修改
- 需要分阶段完成的任务
- 需要记录进度的项目

**使用方法**：
```
/planning-with-files

或者直接说：
"帮我规划一下如何实现设计工作流功能"
```

**优点**：
- 任务清晰，不会遗漏
- 可以随时暂停和恢复
- 记录进度和发现
- 便于团队协作

**生成的文件**：
```
task_plan.md    - 任务拆解和步骤
findings.md     - 开发过程中的发现
progress.md     - 完成进度
```

#### Skill 5：react-best-practices（React 最佳实践）

**功能**：编写 React/Next.js 代码时，自动应用最佳实践

**使用场景**：
- 写 Web 端页面
- 优化性能
- 代码重构
- 学习 React 最佳实践

**触发方式**：
- 自动触发（当你写 React 代码时）

**优点**：
- 避免常见错误（如 useEffect 依赖问题）
- 性能优化（useMemo、useCallback）
- 代码更规范
- 符合官方推荐

**示例**：
```javascript
// AI 会自动应用最佳实践

// ❌ 不好的写法
function UserList() {
  const [users, setUsers] = useState([])

  useEffect(() => {
    loadUsers()
  })  // 缺少依赖数组，会无限循环

  return users.map(user => <div>{user.name}</div>)
}

// ✅ AI 生成的代码
function UserList() {
  const [users, setUsers] = useState([])

  useEffect(() => {
    loadUsers()
  }, [])  // 空依赖数组，只执行一次

  return users.map(user => (
    <div key={user.id}>{user.name}</div>  // 添加 key
  ))
}
```

### 5.3 更多实用 Skills

#### Skill 6：simplify（代码简化和优化）

**功能**：审查代码质量，发现可以优化的地方并自动优化

**使用场景**：
- 代码写完后想优化
- 代码太复杂，想简化
- 想学习更好的写法
- 代码重构

**使用方法**：
```
/simplify

或者直接说：
"帮我优化一下这段代码"
```

**优点**：
- 自动发现重复代码
- 提高代码可读性
- 应用最佳实践
- 减少代码行数

**示例**：
```javascript
// 优化前
function getUserName(user) {
  if (user) {
    if (user.name) {
      return user.name
    } else {
      return '未命名'
    }
  } else {
    return '未命名'
  }
}

// AI 优化后
function getUserName(user) {
  return user?.name || '未命名'
}
```

## 第六部分：AI 常见问题

### 6.1 AI 理解错了需求

#### 问题示例

```
你的需求：
"帮我做一个用户列表，按年龄排序"

AI理解成：
"按年龄升序排序"

实际需求：
"按年龄降序排序"
```

#### 解决方案

```
"不对，我要的是按年龄降序排序，年龄大的在前面。

示例：
- 张三，30岁
- 李四，25岁
- 王五，20岁"
```

### 6.2 AI生成的代码有错

#### 问题示例

```javascript
// AI生成的代码
const res = await db.collection('users').get()
this.setData({ users: res.data })
// 但实际运行报错：res.data is undefined
```

#### 解决方案

```
"这段代码运行报错：res.data is undefined

我打印了 res，发现数据在 res.result.data 里，不是 res.data

请修改代码"
```

**预防方法**：
- 使用 verification-before-completion 验证代码
- 提供数据结构示例
- 先小范围测试

### 6.3 AI修改了不该改的文件

#### 问题示例

```
你的需求：
"修改用户列表页的样式"

AI的操作：
- 修改了 pages/users/index.wxss ✅
- 修改了 app.wxss ❌（不该改）
- 修改了其他页面的样式 ❌（不该改）
```

#### 解决方案

```
"不要修改 app.wxss 和其他页面的样式，只修改 pages/users/index.wxss"
```

**预防方法**：
- 明确指定要修改的文件
- 告诉 AI 不要修改哪些文件
- 使用 Git 随时回退

### 6.4 如何让AI记住规则

#### 方法1：在每次对话中重复

```
"请帮我修改代码，注意：
1. 时间字段统一用 { $date: 毫秒 } 格式
2. 换行符要转义
3. 所有列表都要分页加载
4. 错误提示要友好"
```

#### 方法2：创建规范文档

```
创建一个 RULES.md 文件，写入规范：

# 开发规范

## 数据格式
- 时间字段：{ $date: 毫秒 }
- 换行符：写入前转义，读取后还原

## 性能优化
- 列表分页加载（每页20条）
- 图片压缩后上传
- 批量更新 setData

## 用户体验
- 显示加载状态
- 友好的错误提示
- 空状态设计

然后告诉AI：
"请按照 RULES.md 中的规范开发"
```

#### 方法3：使用项目记忆（如果AI支持）

```
"请记住以下规则，在后续开发中都要遵守：
1. 时间字段统一用 { $date: 毫秒 } 格式
2. 换行符要转义
3. 所有列表都要分页加载"
```

### 6.5 如何提高 AI 效率（省 Token）

#### 技巧1：使用 Grep 而不是 Read

```
❌ 浪费 Token：
"帮我读一下所有页面文件，找出哪里用了 localStorage"
→ AI 会读取所有文件，消耗大量 Token

✅ 高效搜索：
"用 Grep 搜索项目中所有使用 localStorage 的地方"
→ AI 只返回匹配的行，Token 消耗少
```

#### 技巧2：明确指定文件

```
❌ 让 AI 猜：
"帮我改一下客户列表的样式"
→ AI 需要先搜索文件，再读取，再修改

✅ 直接指定：
"修改 pages/leadList/index.wxss，把客户卡片的背景色改为白色"
→ AI 直接修改，省时省 Token
```

#### 技巧3：使用 Edit 而不是 Write

```
❌ 重写整个文件：
"帮我修改 leadList/index.js，加一个搜索框"
→ AI 可能会重写整个文件

✅ 只修改需要的部分：
"在 leadList/index.js 的第 50 行后面加一个搜索框组件"
→ AI 使用 Edit 工具，只发送修改的部分
```

#### 技巧4：批量操作

```
❌ 一个个改：
"帮我把 leadList/index.js 的 localStorage 改成 pnzj_user"
"再帮我把 projectList/index.js 的 localStorage 改成 pnzj_user"
"再帮我把 quoteList/index.js 的..."

✅ 一次性批量：
"用 Grep 搜索所有使用 localStorage.getItem('user') 的地方，
然后批量替换为 localStorage.getItem('pnzj_user')"
```

#### 技巧5：清理上下文

```
当对话很长时，可以清空上下文，重新开始新的任务。

注意：清空后 AI 会忘记之前的对话，
但项目文件和文档中的记录会保留。
```

## 第七部分：最佳实践

### 7.1 分步骤开发

#### 不要一次性要求太多

```
❌ 不好的方式：
"帮我做一个完整的商城小程序，包括：
- 商品列表
- 商品详情
- 购物车
- 订单
- 支付
- 用户中心
- ..."

✅ 好的方式：
"我们分步骤开发商城小程序：

第一步：商品列表页
- 显示商品图片、名称、价格
- 支持分页加载
- 点击跳转详情页

完成第一步后，我们再做第二步"
```

#### 每完成一步就测试

```
1. AI生成代码
   ↓
2. 你测试代码
   ↓
3. 发现问题 → 让AI修复 → 重新测试
   ↓
4. 没问题 → 进行下一步
```

### 7.2 及时反馈

#### 告诉AI结果

```
✅ 好的反馈：
"你生成的代码我测试了，功能正常，但有个小问题：
- 列表加载成功 ✅
- 分页功能正常 ✅
- 但是图片显示不出来 ❌

图片的 fileID 是正确的，但页面显示不出来"

❌ 不好的反馈：
"不行"
```

#### 提供测试结果

```
"我测试了你的代码：

测试1：正常情况
- 数据库有10条数据
- 结果：显示正常 ✅

测试2：空数据
- 数据库没有数据
- 结果：显示空白，没有提示 ❌
- 预期：应该显示'暂无数据'

请修复测试2的问题"
```

### 7.3 建立检查清单

#### 代码检查清单

```
每次AI生成代码后，检查：

□ 语法是否正确（有没有括号、分号错误）
□ 逻辑是否正确（是否符合需求）
□ 数据格式是否正确（字段名、类型）
□ 错误处理是否完善（try-catch、提示）
□ 性能是否优化（分页、批量更新）
□ 用户体验是否友好（加载状态、空状态）
```

#### 测试检查清单

```
每次测试代码时，检查：

□ 正常情况是否正常
□ 空数据是否有提示
□ 网络错误是否有提示
□ 数据格式错误是否处理
□ 边界情况是否考虑
```

### 7.4 记录文档

#### 记录重要决策

```
在项目中创建 DECISIONS.md：

# 技术决策记录

## 2026-04-27：时间字段格式统一
- 问题：时间字段格式不统一，导致排序混乱
- 决策：统一使用 { $date: 毫秒 } 格式
- 原因：兼容性好，易于处理
- 影响：所有新代码都要遵守

## 2026-04-28：列表分页加载
- 问题：一次性加载所有数据，页面卡顿
- 决策：所有列表都要分页加载，每页20条
- 原因：提高性能，改善用户体验
- 影响：所有列表页面都要修改
```

#### 记录踩过的坑

```
在项目中创建 PITFALLS.md：

# 踩坑记录

## 时间字段排序混乱
- 问题：用 String() 转换时间后排序失败
- 原因：对象转字符串变成 "[object Object]"
- 解决：创建统一的时间解析函数
- 教训：不要用字符串比较时间

## Mock数据遗留
- 问题：开发时用Mock数据，上线后忘记替换
- 原因：没有及时删除Mock代码
- 解决：上线前全面检查，删除所有Mock
- 教训：Mock代码要及时删除
```

### 7.5 与 AI 协作的最佳实践

#### 1. 把 AI 当成高级助手，不是魔法师

- AI 很强，但不是万能的
- 复杂的业务逻辑需要你来设计
- AI 负责实现和优化
- 你负责决策和验证

#### 2. 保持对话的连贯性

- 一个功能开发完再开始下一个
- 不要在一个对话中跳来跳去
- 如果要换主题，可以清空上下文重新开始
- 保持任务的聚焦

#### 3. 及时记录重要信息

- 重要的决策写到文档中
- 遇到的坑写到记录中
- 不要依赖 AI 的记忆（会被清空）
- 建立项目知识库

#### 4. 善用版本控制

- 每完成一个功能就提交
- 方便回退和对比
- AI 也可以帮你提交（但要检查）
- 保持提交记录清晰

#### 5. 定期复盘

- 每周回顾一下开发进度
- 总结哪些地方 AI 帮助很大
- 哪些地方需要改进
- 持续优化协作方式
