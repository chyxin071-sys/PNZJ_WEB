# 从需求到PRD：如何把想法变成可执行的开发计划

> 本指南教你如何从一个模糊的想法，通过与AI对话，最终形成清晰的产品需求文档（PRD）和开发计划。

---

## 为什么需要这一步？

很多新手（包括我自己）一开始都是：
```
脑子里有个想法 → 直接开始写代码 → 写着写着发现不对 → 推倒重来
```

**正确的流程应该是**：
```
模糊的想法 → 需求分析 → PRD → 数据库设计 → 开发
```

**为什么要这样？**
- 避免返工（想清楚再做）
- 避免遗漏功能（全面梳理）
- 方便团队协作（统一认知）
- 方便后期维护（有文档可查）

## 第一步：需求收集（把想法说出来）

### 1.1 你现在的状态

你可能只有一个模糊的想法：

```
"我想做一个装修公司管理系统"
"我想做一个外卖小程序"
"我想做一个记账工具"
```

**这还不够！** 你需要把想法展开。

### 1.2 使用 AI 帮你梳理需求

**使用 brainstorming skill**：

> 💡 **Skill 详细介绍**：查看 [05_AI辅助开发指南 - Skill 1：brainstorming](./05_AI_ASSISTED_DEVELOPMENT.md#skill-1brainstorming头脑风暴)

```
你："我想做一个装修公司管理系统，帮我梳理一下需求"

AI：让我帮你梳理需求，我会问你一些问题：

1. 用户角色
   - 谁会用这个系统？（老板、销售、设计师、项目经理？）
   - 每个角色需要什么功能？

2. 核心功能
   - 客户管理：需要记录哪些信息？
   - 工地管理：需要跟踪哪些进度？
   - 报价管理：报价流程是怎样的？

3. 业务流程
   - 从客户咨询到签单的流程是什么？
   - 从签单到施工完成的流程是什么？

4. 数据管理
   - 需要存储哪些数据？
   - 数据之间的关系是什么？

5. 特殊需求
   - 需要通知功能吗？
   - 需要权限控制吗？
   - 需要数据统计吗？
```

### 1.3 回答 AI 的问题

**一步步回答，不要着急**：

```
你："用户角色有4种：
1. 老板（查看所有数据、数据统计）
2. 销售（管理客户、跟进、报价）
3. 设计师（接收设计任务、上传设计图）
4. 项目经理（管理工地进度、验收）"

AI："好的，那我们来细化每个角色的功能..."

你："客户管理需要记录：
- 基本信息（姓名、电话、地址）
- 客户来源（朋友介绍、广告、自然流量）
- 跟进记录（每次沟通的内容）
- 客户状态（待跟进、已报价、已签单）"

AI："明白了，那跟进记录需要记录哪些内容？"

...（继续对话）
```

### 1.4 需求收集的技巧

**技巧1：从用户角度思考**
```
不要想"我要做什么功能"
而是想"用户要完成什么任务"

❌ "我要做一个客户列表"
✅ "销售需要快速找到待跟进的客户"
```

**技巧2：想象真实场景**
```
想象一个销售的一天：
- 早上：查看今天要跟进的客户
- 上午：打电话跟进，记录沟通内容
- 下午：给客户发报价
- 晚上：查看今天的工作总结

→ 从这些场景中提取功能需求
```

**技巧3：不要一开始就想太复杂**
```
第一版：只做核心功能
第二版：再加辅助功能
第三版：优化体验

不要想着一次做完所有功能！
```

## 第二步：需求分析（整理成结构化的需求）

> 📚 **相关章节**：
> - 数据库设计原则：[01_小程序完全指南 - 铁律1：先设计数据库结构](./01_MINIPROGRAM_COMPLETE_GUIDE.md#铁律1先设计数据库结构最重要)
> - 时间格式统一：[01_小程序完全指南 - 铁律2：统一时间格式](./01_MINIPROGRAM_COMPLETE_GUIDE.md#铁律2统一时间格式设计阶段就要定)

### 2.1 用户角色与权限

| 角色 | 权限 | 主要功能 |
|------|------|---------|
| 老板 | 全部 | 查看所有数据、数据统计、员工管理 |
| 销售 | 客户相关 | 客户管理、跟进记录、报价 |
| 设计师 | 设计相关 | 查看设计任务、上传设计图 |
| 项目经理 | 工地相关 | 工地进度管理、验收 |

### 2.2 核心功能列表

**客户管理（CRM）**
- 客户列表（筛选、搜索）
- 客户详情（基本信息、跟进记录）
- 新增客户
- 编辑客户
- 跟进记录（新增、查看）
- 客户状态变更

**报价管理**
- 报价列表
- 创建报价（选择材料、计算价格）
- 查看报价详情
- 发送报价给客户

**工地管理**
- 工地列表
- 工地详情（进度、照片）
- 进度更新
- 工序验收
- 工地照片上传

**员工管理**
- 员工列表
- 新增员工
- 编辑员工
- 权限设置

**通知系统**
- 任务分配通知
- 进度更新通知
- 验收通知

### 2.3 业务流程图

**客户签单流程**：
```
1. 销售新增客户
   ↓
2. 销售跟进客户（记录沟通内容）
   ↓
3. 客户有意向 → 销售创建报价
   ↓
4. 客户确认报价 → 销售标记"已签单"
   ↓
5. 系统自动创建工地
   ↓
6. 通知项目经理
```

**工地施工流程**：
```
1. 项目经理接收工地
   ↓
2. 开工（记录开工时间）
   ↓
3. 按工序施工（拆除、水电、泥瓦...）
   ↓
4. 每个工序完成后验收
   ↓
5. 全部完成 → 竣工验收
```

### 2.4 数据结构设计

**客户表（leads）**
```javascript
{
  _id: "自动生成",
  name: "客户姓名",
  phone: "手机号",
  address: "地址",
  source: "客户来源",
  status: "待跟进 | 已报价 | 已签单",
  salesPerson: "负责销售",
  createdAt: 时间戳,
  updatedAt: 时间戳
}
```

**跟进记录表（followUps）**
```javascript
{
  _id: "自动生成",
  leadId: "客户ID",
  content: "跟进内容",
  createdBy: "创建人",
  createdAt: 时间戳
}
```

**工地表（projects）**
```javascript
{
  _id: "自动生成",
  leadId: "客户ID",
  address: "工地地址",
  status: "未开工 | 施工中 | 已完工",
  manager: "项目经理",
  startDate: 时间戳,
  endDate: 时间戳,
  createdAt: 时间戳
}
```

## 第四步：要不要做原型？

### 4.1 什么时候需要原型

**需要做原型**：
- ✅ 团队项目（多人协作，需要统一认知）
- ✅ 复杂的交互（流程复杂，需要可视化）
- ✅ 给客户看（需要演示效果）
- ✅ 不确定UI怎么做（需要先画出来看看）

**可以不做原型**：
- ✅ 个人项目（自己做，脑子里有画面）
- ✅ 简单的CRUD（列表、表单，很常见）
- ✅ 参考现有产品（照着别人的做）
- ✅ 快速验证想法（先做出来再说）

### 4.2 原型的几种形式

**1. 纸笔草图（最快）**
```
拿张纸，画出页面布局
- 顶部是什么
- 中间是什么
- 底部是什么
- 点击后跳转到哪里

优点：快，5分钟搞定
缺点：不够精细
```

**2. 文字描述（适合AI）**
```
"客户列表页：
- 顶部：搜索框
- 中间：客户卡片列表
  - 每个卡片显示：姓名、电话、状态
  - 点击卡片跳转到详情页
- 底部：新增客户按钮"

优点：清晰，AI能理解
缺点：没有视觉效果
```

**3. 原型工具（最专业）**
```
使用工具：Figma、Sketch、墨刀、即时设计

优点：专业，可以演示
缺点：学习成本高，耗时
```

**4. 直接写代码（最直接）**
```
不做原型，直接写小程序代码
边写边调整UI

优点：快，直接看到效果
缺点：改起来麻烦
```

### 4.3 我的建议

**对于小程序开发**：

**简单项目**（如记账工具、待办事项）：
```
不需要原型，直接写代码
或者画个纸笔草图（5分钟）
```

**中等项目**（如本项目的装修管理系统）：
```
用文字描述 + 参考现有产品
"客户列表参考微信通讯录的样式"
"工地详情参考淘宝订单详情的布局"
```

**复杂项目**（如电商、社交）：
```
建议做原型（至少低保真原型）
使用原型工具画出主要页面
```

### 4.4 使用 AI 生成原型代码

**即使不做原型，也可以让AI帮你快速搭建UI**：

```
你："帮我做一个客户列表页，要求：
- 顶部搜索框
- 中间客户卡片列表（姓名、电话、状态）
- 底部新增按钮
- 参考微信通讯录的样式"

AI："好的，我来写代码..."
[生成 wxml + wxss + js]

你："卡片太挤了，加点间距"

AI："好的，我来调整..."
```

**这样就相当于"边做原型边开发"**。

## 常见问题

### Q1：PRD一定要写得很详细吗？

**A**：看项目大小

**小项目**（个人开发）：
- 简单写写就行，主要是理清思路
- 重点是功能列表和数据库设计

**大项目**（团队协作）：
- 需要详细的PRD
- 包括用户故事、交互细节、边界情况

### Q2：PRD写完后能改吗？

**A**：当然可以改！

```
PRD不是一成不变的
开发过程中发现问题，随时更新PRD

但要注意：
- 大的改动要评估影响
- 改动要同步给团队
- 改动要记录（版本管理）
```

### Q3：不写PRD直接开发可以吗？

**A**：可以，但不推荐

```
不写PRD的后果：
- 容易遗漏功能
- 容易返工
- 后期维护困难（忘了当初为什么这么做）

至少要有：
- 功能列表
- 数据库设计
```

### Q4：AI能帮我写PRD吗？

**A**：可以，但需要你提供足够的信息

```
好的做法：
1. 先和AI对话，梳理需求
2. 让AI根据对话内容生成PRD
3. 你再检查和补充

不好的做法：
直接说"帮我写个记账小程序的PRD"
→ AI写出来的会很空洞
```

**从需求到PRD的完整流程**：

```
1. 需求收集
   - 把想法说出来
   - 使用 brainstorming skill
   - 回答AI的问题

2. 需求分析
   - 整理用户角色
   - 列出核心功能
   - 画出业务流程
   - 设计数据结构

3. 编写PRD
   - 使用PRD模板
   - 详细描述每个功能
   - 列出开发计划

4. 原型设计（可选）
   - 简单项目：不需要
   - 复杂项目：建议做

5. 开始开发
   - 先设计数据库
   - 按优先级开发
```

**关键要点**：
- 不要跳过需求分析，直接写代码
- 使用AI帮你梳理需求（brainstorming）
- PRD不需要完美，但要清晰
- 数据库设计最重要，一定要想清楚
