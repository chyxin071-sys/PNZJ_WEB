# CM1.0 问题清单 & 风险评估

> 版本：1.0 | 更新日期：2026-04-17

---

## 一、逻辑未闭环 / 数据未联通

### 1. 合同模块（contracts）完全是假数据
- **位置**：`web/src/app/contracts/[id]/page.tsx`
- **问题**：合同详情页的客户信息、付款节点全部是硬编码的 mock 数据，没有从数据库读取。合同状态（待上传/执行中/已结项）是用 `id % 3` 这种数学运算模拟的，不是真实状态字段。
- **影响**：合同管理功能对用户来说是"假的"，无法实际使用。
- **修复方向**：在 `leads` 集合里增加 `contractStatus`、`payments` 字段，或单独建 `contracts` 集合。

### 2. 报价单与线索/项目没有关联
- **位置**：`web/src/app/api/quotes/route.ts`、`web/src/models/Lead.ts`
- **问题**：报价单（quotes）里没有 `leadId` 或 `projectId` 字段，无法从线索详情页直接跳转到对应报价单，也无法从报价单反查是哪个客户的。
- **影响**：销售做完报价单，项目经理看不到；线索转项目时报价信息断链。
- **修复方向**：quotes 集合增加 `leadId`、`projectId` 字段；创建报价单时必须关联线索。

### 3. 线索转项目没有自动联通
- **位置**：`web/src/app/leads/page.tsx`（签单逻辑）、`web/src/app/api/projects/route.ts`
- **问题**：线索签单后状态变为"已签单"，但没有自动创建对应的项目记录，也没有把 `leadId` 写入 project。项目详情页里的 `quoteId` 是手动查询的，不是从 lead 关联过来的。
- **影响**：签单后需要手动再去新建项目，容易遗漏，且两边数据不同步（客户名、地址等）。
- **修复方向**：签单时自动触发创建项目草稿，或至少在 project 里存 `leadId`。

### 4. 项目 Model 与实际数据结构不一致
- **位置**：`web/src/models/Project.ts` vs `web/src/app/api/projects/route.ts`
- **问题**：Mongoose Model 里的 Project 字段（`customer`, `manager`, `stage` 等）与 TCB 云数据库里实际存储的字段（`nodesData`, `startDate`, `sales`, `designer` 等）完全不同。Model 文件实际上没有被使用，只是一个废弃的定义。
- **影响**：代码误导性强，维护时容易混淆。

### 5. 员工管理 PATCH 接口引用了未导入的 `tcbQuery`
- **位置**：`web/src/app/api/employees/[id]/route.ts` 第 2 行只导入了 `tcbUpdate, tcbDelete`，但第 16 行调用了 `tcbQuery`
- **问题**：这是一个运行时 bug，当管理员修改员工姓名时，获取旧名字的步骤会报 `tcbQuery is not a function` 错误，导致全局名字同步失败。
- **影响**：管理员在网页端修改员工姓名后，关联数据不会同步更新。
- **修复**：在 import 里加上 `tcbQuery`。

### 6. 密码安全问题（明文存储）
- **位置**：`web/src/app/api/employees/route.ts`（第 63 行）、`web/src/app/api/auth/password/route.ts`（第 35 行）、小程序 `pages/profile/index.js`（第 265 行）
- **问题**：新建员工时密码以 `passwordPlain` 明文存入数据库；修改密码时也是存明文；小程序登录时直接比对明文密码。
- **影响**：数据库泄露即等于所有账号密码泄露，是严重安全漏洞。
- **修复方向**：统一使用 bcrypt 哈希存储，登录和修改密码都走 bcrypt.compare。

### 7. 登录接口硬编码了 AppSecret
- **位置**：`web/src/app/api/auth/login/route.ts` 第 6-8 行、`web/src/app/api/upload/route.ts` 第 3-5 行、`web/src/lib/wechat-tcb.ts` 第 3-5 行
- **问题**：APPID、APPSECRET、ENV 直接写在代码里，提交到 git 后任何有代码权限的人都能看到。
- **影响**：AppSecret 泄露可以伪造任意请求，是严重安全风险。
- **修复方向**：移入 `.env.local` 环境变量，代码里用 `process.env.WECHAT_APPSECRET` 读取。

### 8. 数据查询没有分页，硬限制 100 条
- **位置**：所有 GET 接口（leads/projects/todos/quotes/notifications 等）
- **问题**：所有列表查询都是 `.limit(100)` 或 `.limit(200)`，超过这个数量的数据永远查不到。
- **影响**：业务增长后数据会"消失"，用户以为没有数据实际上是查不到。
- **修复方向**：增加分页参数（`page`, `pageSize`），或至少把 limit 提高到 500 并加注释说明。

### 9. 客户编号生成存在并发冲突风险
- **位置**：`web/src/app/api/leads/route.ts` 第 22-36 行
- **问题**：生成客户编号的逻辑是"查最大编号+1"，如果两个请求同时到达，会生成相同的编号。
- **影响**：并发录入时可能出现重复客户编号。
- **修复方向**：使用原子计数器，或在编号后加随机后缀，或改用时间戳+随机数组合。

### 10. 导入功能的客户编号用随机数
- **位置**：`web/src/app/api/import/route.ts` 第 21 行
- **问题**：批量导入时客户编号是 `P${year}IMP${随机3位数}`，可能重复，且格式与正常编号不一致。
- **影响**：导入的数据编号混乱，无法按编号追溯。

### 11. 网页端 profile 页面统计数据逻辑有误
- **位置**：`web/src/app/profile/page.tsx` 第 68-70 行
- **问题**：统计"我的待办"时用的是 `t.assignedTo?.name`，但 todos 数据结构里是 `assignees`（数组），导致统计永远为 0。
- **影响**：个人主页的待办数量显示不准确。

### 12. 通知系统没有自动触发
- **位置**：整个系统
- **问题**：通知（notifications）只能手动创建，没有任何业务事件自动触发通知。例如：线索分配给销售、项目节点完成、待办到期等，都不会自动发通知。
- **影响**：通知功能形同虚设，用户需要手动发消息。

### 13. 小程序端修改密码只存明文，不走 API
- **位置**：`mini-program/pages/profile/index.js` 第 264-268 行
- **问题**：小程序直接操作云数据库更新 `passwordPlain`，绕过了网页端的 `/api/auth/password` 接口，两套逻辑不一致。
- **影响**：密码修改逻辑分裂，且都是明文存储。

## 二、边界校验缺失

| 位置 | 缺失的校验 |
|------|-----------|
| 新建线索 | 手机号格式校验（11位数字）、姓名长度限制 |
| 新建项目 | 开工日期不能早于今天、面积必须为正数 |
| 报价单 | 金额不能为负数、单价×数量=小计的一致性校验 |
| 员工管理 | 手机号格式校验、密码强度要求（最少6位） |
| 待办 | 截止日期不能早于创建日期 |
| 项目节点 | 工期天数必须为正整数 |
| 文件上传 | 文件大小限制、文件类型白名单 |

## 三、验收标准（1.0 版本）

以下是判断 1.0 版本"可以正式使用"的最低标准：

### 必须修复（上线前）
- [ ] 员工 PATCH 接口 `tcbQuery` 未导入的 bug
- [ ] 密码明文存储改为 bcrypt 哈希
- [ ] AppSecret 移入环境变量
- [ ] 网页端 profile 待办统计字段名修正

### 应该修复（1.1 版本前）
- [ ] 合同模块接入真实数据
- [ ] 报价单增加 leadId 关联字段
- [ ] 通知系统增加关键业务事件自动触发
- [ ] 数据查询分页或提高 limit

### 可以延后（1.x 版本）
- [ ] 线索转项目自动联通
- [ ] 客户编号并发冲突修复
- [ ] 导入功能编号规范化
