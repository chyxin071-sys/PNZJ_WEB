# CM1.0 问题清单 & 修复优先级

> 整理日期：2026-04-22 | 来源：用户测试反馈

---

## 优先级说明

- **P0 - 紧急**：影响核心业务流程，必须立即修复
- **P1 - 高**：影响日常使用体验，本周内修复
- **P2 - 中**：体验优化，下个迭代修复
- **P3 - 低**：细节打磨，排期处理
- **❓ - 待调研**：需要进一步确认方案或原因

---

## P0 - 紧急（阻断业务）

### BUG-01 客户/工地下拉无法更新（修改名字后其他地方未同步）
- **现象**：修改客户或工地名字后，其他关联页面的下拉选项、显示名称没有同步更新
- **影响范围**：全系统关联字段
- **修复方向**：检查 employees PATCH 接口的 `tcbQuery` 未导入 bug（已记录在 ISSUES_AND_RISKS.md #5），同时排查 leads/projects 名字变更后的级联更新逻辑
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：`employees/[id]/route.ts` 的名字同步查询中补充了 `leads.manager` 字段的同步

### BUG-02 编辑工地信息总是失败
- **现象**：编辑工地信息时点击保存，提示失败或无响应
- **影响范围**：工地管理核心功能
- **修复方向**：排查 `/api/projects/[id]` PATCH 接口，检查请求体字段名是否与数据库字段一致，检查项目经理字段类型（见 BUG-14）
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：
  1. 根本原因：所有 API 的 `docData` 构建时 `.replace(/\\n/g, '\\\\n')` 正则写错，实际上没有转义真正的换行符，导致跟进记录内容含换行时 TCB 查询语句解析失败。已修正为 `.replace(/\n/g, '\\n')`，影响 6 个 API 文件
  2. `projects/[id]/page.tsx` 读取 localStorage 时用的是错误的 key `'user'`，已改为 `'pnzj_user'`

### BUG-03 项目经理无法关联客户创建工地
- **现象**：客户详情只有销售和设计师字段，项目经理没有关联入口，导致项目经理无法从客户创建工地
- **修复方向**：客户详情页增加"项目经理"字段（可多选），与销售/设计师并列
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：`leads/[id]/page.tsx` 中项目经理卡片从仅"已签单/施工中"状态显示，改为始终显示，文案从"待指派"改为"未分配"，按钮从"指派"改为"分配"

## P1 - 高（影响日常使用）

### BUG-04 表单中"指派"用词问题
- **现象**：新建/编辑表单中出现"指派销售""指派设计师""指派执行人"等措辞
- **修复方向**：全局替换，改为直接显示"销售""设计师""执行人（可多选）"，去掉"指派"二字
- **端**：网页端 + 小程序

### BUG-05 待办缺少"关联客户"筛选功能
- **现象**：待办列表没有按关联客户筛选的入口
- **修复方向**：待办页面筛选栏增加"关联客户"下拉筛选
- **端**：网页端 + 小程序

### BUG-06 跟进记录排序不一致（时有正序时有倒序）
- **现象**：改状态时系统自动添加的跟进记录是倒序，完成待办后添加的又变成正序，显示混乱
- **修复方向**：统一按时间倒序（最新在最上面）显示所有跟进记录，排查各处添加跟进记录后的列表刷新逻辑
- **端**：网页端 + 小程序

### BUG-07 开启设计流无通知
- **现象**：开启设计流后，相关设计师和管理员没有收到通知
- **修复方向**：开启设计流时，自动向关联设计师 + 所有 admin/manager 发送系统通知
- **端**：网页端，小程序

### BUG-08 开启工作流的系统跟进记录内容混乱
- **现象**：开始工作流时系统自动添加的跟进记录内容没有换行，信息堆在一起难以阅读
- **修复方向**：跟进记录内容使用换行符分隔各项信息（如节点名称、时间、负责人等）
- **端**：网页端，小程序

### BUG-09 设计进度非设计师也能编辑
- **现象**：设计进度模块对所有角色都可编辑
- **修复方向**：设计进度编辑权限限制为 `designer` 角色，其他角色只读
- **端**：网页端，小程序

### BUG-10 无关人员可以查看客户的项目资料
- **现象**：与该客户无关联的员工进入客户详情，仍能看到项目资料（工地、报价单等）
- **修复方向**：未签单客户的项目资料，仅限关联的销售/设计师/项目经理及 admin 可见；已签单客户信息全员可见（只读）
- **端**：网页端，小程序

### BUG-11 修改状态时下拉菜单默认不在当前状态
- **现象**：客户当前状态是"已量房"，点开状态修改菜单时默认选中的是第一项而不是"已量房"
- **修复方向**：状态选择器打开时，默认高亮/选中当前状态值；排查其他类似下拉（工地节点状态、待办状态等）是否有同样问题
- **端**：网页端 + 小程序

### BUG-12 新建客户时无系统跟进记录
- **现象**：新建客户后，跟进记录里没有"客户创建"的系统通知
- **修复方向**：新建客户时自动写入一条系统跟进记录，内容包含创建人、创建时间、初始状态
- **端**：网页端，小程序

### BUG-13 签单时未通知全员
- **现象**：客户签单后没有全员消息通知（之前改过，需确认小程序是否已更新）
- **修复方向**：确认网页端签单通知逻辑是否正常；小程序需重新发布更新
- **端**：网页端 + 小程序（❓ 先确认是否是小程序版本未更新导致）

### BUG-14 编辑工地时项目经理是输入框而非选择器
- **现象**：编辑工地信息时，项目经理字段是手动输入姓名，而不是从员工列表选择
- **修复方向**：改为从 users/employees 集合下拉选择，存储 `{_id, name}` 结构
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：`projects/[id]/page.tsx` 编辑弹窗中项目经理改为从 employees 接口加载的 manager/admin 角色员工下拉选择，同时开工日期不再强制要求项目经理必填

### BUG-15 从工地详情点进客户信息一直转圈
- **现象**：在工地详情页点击客户信息链接，页面一直加载转圈，无法显示
- **修复方向**：排查工地详情页跳转客户详情的 leadId 传参是否正确，检查客户详情页的数据加载逻辑
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：`leads/[id]/page.tsx` 的 `fetchLeadDetail` 在 API 返回非 200 时没有退出，导致 `lead` 永远是 null 页面一直转圈。已加入 `leadNotFound` 状态，API 失败时显示"找不到该客户"提示页

### BUG-16 报价单保存后跳出页面
- **现象**：保存报价单后页面跳转离开，应该留在当前报价单页面
- **修复方向**：保存成功后不跳转，刷新当前页数据即可
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：修改 `quoteDetail/index.js` 和 `web` 端保存逻辑，不再直接 `router.replace` 或 `wx.navigateBack()`，而是提示保存成功并更新当前页面状态。

### BUG-17 报价单每次打开都是新建状态
- **现象**：点开报价单默认显示空白新建状态，而不是上一个已保存的版本
- **修复方向**：进入报价单页面时，优先加载该客户/工地最新的已保存报价单版本
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：修改 `checkQuoteStatus` 的拉取逻辑增加按创建时间倒序获取最新，网页端在 `quotes/new` 拦截检测已存在则跳转详情。

### BUG-18 第一次创建报价单保存时弹出"覆盖还是新建"
- **现象**：第一次创建报价单点保存，弹出覆盖/新建的选择弹窗，逻辑不合理
- **修复方向**：检测是否存在已有版本，第一次保存直接保存，不弹确认框
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：小程序端引入 `isNewQuote` 标记，首次创建空白报价单点击保存时不再弹出覆盖提示，直接静默更新。

## P2 - 中（体验优化）

### BUG-19 销售/设计师选择器排序混乱
- **现象**：选择销售和设计师时，人员列表顺序混乱，管理员排在前面
- **修复方向**：人员选择器中，销售/设计师角色排在前面，manager/admin 排在后面
- **端**：网页端 + 小程序

### BUG-20 新建待办执行人未默认为自己
- **现象**：新建待办时，执行人字段为空，需要手动选择自己
- **修复方向**：新建待办时，执行人默认填入当前登录用户
- **端**：网页端 + 小程序
- **✅ 已修复（2026-04-22）**：在打开新建待办弹窗时将当前用户的 ID 默认放入 assignees 中。

### BUG-21 新建客户时未根据创建人身份自动关联
- **现象**：销售新建客户后，还需要手动把自己添加为该客户的销售；设计师同理
- **修复方向**：新建客户时，根据创建人角色自动填入对应字段（销售→销售字段，设计师→设计师字段，项目经理→项目经理字段）
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：网页端 `handleAddLead` 和小程序表单逻辑中新增对 manager 角色的自动识别填充。

### BUG-22 已签单客户信息应全员可见（只读）
- **现象**：客户签单后，无关人员仍无法查看该客户的完整信息（跟进记录、设计流、工地、资料、报价单等）
- **修复方向**：客户状态变为"已签单"后，全员可查看所有信息，但无关人员无编辑权限
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：在 `leadDetail` 增加 `isVisible` 判断。

### BUG-23 报价单更新未自动添加跟进记录
- **现象**：报价单保存/更新后，客户跟进记录里没有对应的系统记录
- **修复方向**：报价单保存时自动写入跟进记录，内容包含版本号、操作人、时间
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：双端已增加 `addSystemFollowUpToLead`，写入报价单总价和修改人等详细信息。

### BUG-24 新建工地未自动添加跟进记录
- **现象**：新建工地后，客户跟进记录里没有"工地创建"的系统记录
- **修复方向**：新建工地时自动写入跟进记录，显示预计开工时间和预计完工时间
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：确认开工时间后调用系统写入。

### BUG-25 客户状态来源选项修改
- **现象**：原有的“小红书”、“大众点评”不常用，选择“其他”时无处填写详情。
- **修复方向**：移除小红书/大众点评选项，新增其他来源输入框，并支持存入对应文本。
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：更新了两端的 `sourceOptions` 并在选“其他”时展示附加的文本输入框。

### BUG-26 待办筛选关联客户泄露隐私
- **现象**：非管理员可在待办的高级筛选中看见全公司打码客户的全名。
- **修复方向**：普通员工在“关联客户”的下拉列表中只能看到自己相关或已签单客户。
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：筛选出 `filteredLeads` 供下拉菜单使用，避免越权。

### BUG-27 小程序现场影像布局问题
- **现象**：上传的现场影像不是 3×3 排列，且与上方框体左右不对齐
- **修复方向**：改为 3 列 grid 布局，与上方容器左右边距对齐，做响应式设计
- **端**：小程序，小程序
- **✅ 已修复（2026-04-22）**：`projectDetail/index.wxml` 将验收记录图片的 grid-template-columns 改为 repeat(3, 1fr)，添加 padding: 0 20rpx 对齐容器

### BUG-28 材料库有重复数据
- **现象**：材料库中存在重复的材料条目
- **修复方向**：排查数据库中的重复记录，前端增加重名校验提示
- **端**：网页端，小程序
- **✅ 已修复（2026-04-22）**：`materialForm/index.js` 在保存材料时添加重名检查，防止新增重复材料

### BUG-29 网页端云托管频繁 503
- **现象**：网页端云托管（GitHub 部署）经常出现 503 错误
- **可能原因**：频繁推送代码触发持续部署，部署期间服务不可用；或部署失败导致服务异常
- **建议**：
  1. 检查微信云托管控制台的部署日志，确认是否有失败记录
  2. 考虑合并多次小改动后再推送，减少部署频率
  3. 配置部署失败告警
- **端**：网页端（❓ 待确认是部署中断还是部署失败）
- **⏸️ 暂不处理**：待所有BUG修复完成后重新部署时检查

### BUG-30 数据导出缺少跟进记录
- **现象**：从数据库导出时只能导出客户基本信息，跟进记录没有导出
- **修复方向**：导出功能增加跟进记录字段，或提供单独的跟进记录导出选项
- **端**：网页端，小程序（❓ 待确认导出功能的实现位置）
- **✅ 已确认需求**：需要在云开发后台导出时包含跟进记录字段，待数据库结构调整

### BUG-31 图片上传速度慢
- **现象**：小程序上传现场影像速度很慢
- **可能原因**：微信云存储 COS 上传本身有速度限制；网络环境影响；文件未压缩直接上传
- **建议**：上传前对图片进行压缩（微信小程序 `wx.compressImage`），可显著提升速度
- **端**：小程序（❓ 待确认是否已做图片压缩）
- **✅ 已修复（2026-04-22）**：`projectDetail/index.js` 在 chooseMedia 成功后对图片进行压缩（quality: 80），压缩失败则使用原图

### BUG-32 消息通知是否推送到微信
- **现象**：网页端更新数据后，是否会通知到微信（小程序消息/微信服务通知）
- **说明**：目前系统通知仅在 app 内（notifications 页面），不推送到微信。如需微信推送，需要接入微信订阅消息（subscribeMessage）接口，需要用户授权订阅
- **端**：❓ 待确认需求，是否要做微信订阅消息推送
- **⏸️ 暂不处理**：待前面的功能稳定后再考虑实现微信订阅消息推送
