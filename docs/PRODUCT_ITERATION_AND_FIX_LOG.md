# 产品演进、迭代与修复全景日志 (Product Journey & Fix Log)

> 本文档不仅记录了系统每一次的 Bug 修复与体验优化，更提炼了背后的产品思考与战略演进（从 Web 端到小程序的重心转移）。
> 按照版本与时间倒序排列，系统性地梳理了“战略方向 -> 发现问题 -> 分析原因 -> 落地解决”的完整工作留痕，体现产品闭环思维。

## 🚀 V1.2.0 深水区重构与双端协同（近期：2026-04-24 至今）
> **阶段思考**：真实的业务场景比初期设想的更复杂（如：工序不是单行道而是并行的、主材价格不能让客户看到实时变动、公司增加了财务角色）。在这个阶段，我们对大节点流转和主材分享进行了深度的解耦重构。

### 📅 2026-04-27：通知体系重构与全站文件展示升级

#### 1. 微信体验版订阅消息无法送达
- **问题现象与原因**：在手机上测试体验版时，微信订阅消息发送失败。原因是微信的订阅消息接口默认向“正式版”发送，而在开发版/体验版测试时，如果代码未显式指定环境，微信服务器会直接拦截。
- **解决思路与方案**：全面增加环境适配参数。在调用云函数 `sendSubscribeMessage` 时显式传递 `miniprogramState`（'developer' 或 'trial'），确保各环境均能正常触达。
- **产品价值**：打通了测试到生产的完整通知链路，避免因环境问题导致的漏通知。

#### 2. 主材清单图片/缩略图无法加载
- **问题现象与原因**：员工上传到主材清单的图片在列表和详情页无法加载，且客户在分享页看到的也是损坏的图片。原因是前期开发遗留了 mock 假上传代码，生成的仅是本地临时路径，未真正入库。
- **解决思路与方案**：彻底重构主材清单上传逻辑，接入微信真实的 `wx.cloud.uploadFile` 云存储上传，将实际的 `fileID` 存入数据库。
- **产品价值**：确保多端、多角色的文件共享与访问稳定性，消除了死链接。

#### 3. 全站文件与图纸展示体验不佳
- **问题现象与原因**：项目资料、设计工作流中上传的非图片文件（如 PDF/Word/Excel/CAD图纸）显示为单调的默认图标或 Emoji，无法直观辨认格式；图片也未能直接展示缩略图。
- **解决思路与方案**：在公用 `format.wxs` 中增加文件后缀提取与图标映射逻辑。统一实现了**图片直接显示真实缩略图**，非图片文件根据格式分配**专属高亮颜色与文字标识**（如 PDF 为红色，Excel 为绿色，Word 为蓝色，DWG 为紫色）。
- **产品价值**：极大地提升了员工和客户在查阅大量工程资料时的视觉辨识度和找文件效率。

#### 4. 全局通知范围与系统跟进记录缺失
- **问题现象与原因**：报价单更新、设定设计工作流、主材发送确认、查看工地申请等重要业务动作，只通知了单人或未通知，且客户详情中缺乏历史轨迹（跟进记录）留存。
- **解决思路与方案**：将全站核心动作的通知范围规范化为“相关人员（销售、设计、项目经理、创建人） + 所有系统管理员”，并同步补齐所有操作的 `followUps` 写入动作。更新 `NOTIFICATION_RULES.md` 文档。
- **产品价值**：强化了团队内部协同的透明度，确保任何客户的流转均有迹可循。

### 📅 2026-04-26：主材清单深度优化与权限打磨

#### 1. 微信订阅消息重复推送（防打扰优化）
- **问题现象与原因**：系统管理员或员工在使用微信分身进行多角色测试时，经常一次操作收到多条相同的微信订阅弹窗。
- **解决思路与方案**：在云端增加按 `wechatOpenId` 的二次物理去重机制（利用 Set 过滤），并引入了批量处理（Batch Processing），确保一个微信实体账号针对同一事件只会被推送一次。
- **产品价值**：保护用户免受消息轰炸，提升了微信服务通知的专业度和用户体验。

#### 2. 施工人员获取入户密码不便
- **问题现象与原因**：施工人员需要入户密码才能进场，但过去该信息散落在聊天记录或备注中，查找极度不便。
- **解决思路与方案**：在客户资料和工地详情卡片中新增专属的 `doorPassword`（入户密码）字段，替代掉工地卡片中冗余的“预计完工”字段，且严格限制**仅对相关负责人员和管理员可见**。
- **产品价值**：直击现场施工痛点，提高了进场效率，同时兼顾了业主隐私安全。

#### 3. 主材清单录入易错且客户端体验不佳
- **问题现象与原因**：主材录入表单千篇一律，卡片展示层次不清晰，存在漏填风险；且分享给客户后，客户无法确认。
- **解决思路与方案**：
  - 为“其他”分类添加智能提示（如踢脚线提示“是否不同色”，水槽提示“单双槽”）。
  - 优化卡片布局（根据内容动态调整 `margin-bottom` 消除空白行），突出品牌、型号等核心信息。
  - 新增全局多选发送功能，员工可一次性发送多个主材分类给客户确认。
  - 引入快照机制，客户确认状态与员工端数据解耦，并自动触发微信和站内信通知。
- **产品价值**：降低了员工录入心智负担，使呈现给客户的交付物更加专业美观，并实现了客户确认的闭环流转。

#### 4. UI 细节打磨：SVG 图标丢失
- **问题现象与原因**：项目资料和个人中心里的部分编辑图标（SVG）在某些机型上不可见。
- **解决思路与方案**：将内联的 inline backgrounds 提取重构为 `wxss` 样式类进行渲染。

### 📅 2026-04-25：核心流程闭环、安全与 UI 规范化

#### 1. 资料与文件管理模块增强
- **问题现象与原因**：项目资料操作繁琐，设计进度附件删除会误删物理文件，且缺乏操作流转记录。
- **解决思路与方案**：
  - 隐藏重命名和删除按钮，统一收口至“编辑”图标。
  - 解耦文件删除逻辑，设计进度删除附件仅移除引用，不影响资料库。
  - 补全文件流转相关的业务通知和系统跟进记录。
- **产品价值**：保护了文件资产安全，使文件操作更可溯源。

#### 2. 弹窗嵌套与 WXML 渲染白屏
- **问题现象与原因**：页面偶尔白屏或报错 `end tag missing`，弹窗组件被错误嵌套导致内部点击事件（如重命名）失效。
- **解决思路与方案**：排查并补全未闭合的 WXML 标签；重构了页面（如 `projectFiles`）的弹窗层级，解除了错误的组件包裹。

#### 3. 关键工期与计划天数计算异常
- **问题现象与原因**：设计节点的“计划天数”未显示（原代码误用 `item.duration`）；排期计算会错误地把周末剔除；工序验收后没有严格“完成一项开一项”。
- **解决思路与方案**：
  - 替换为 `{{ganttUtil.getPlanDays(item.startDate, item.endDate)}}` 动态计算。
  - 排期天数统一按自然日计算。
  - 修复 `recalculateGantt` 顺延逻辑，仅将第一个未完成工序置为 current。

#### 4. 待办与通知体系防漏与 UI 截断
- **问题现象与原因**：小程序消息列表右侧时间被截断；工地健康状态文案不准确；待办列表关联客户权限存在泄露风险。
- **解决思路与方案**：
  - 规范 CSS 盒模型（`box-sizing: border-box`）修复截断。
  - 将工地健康状态由“严重延期”改为更客观的“逾期”，并实现基于预计完工日期的自动判断。
  - 修复 `todoForm` 关联客户的筛选逻辑（统一为自己相关的客户+已签单客户），修补权限漏洞。

#### 5. 工地进度分享与审核增强
- **问题现象与原因**：访客申请查看工地进度后，员工容易漏审，且审批操作缺乏留痕。
- **解决思路与方案**：
  - 在“查看申请”按钮右上角增加常驻的“待审核红点”提示。
  - 申请和审批动作均自动记录到系统跟进中，并下发双向通知。
  - 列表新增展示“审批时间”和“审批人”。

#### 6. 多端登录冲突与安全问题
- **问题现象与原因**：同一账号可多设备同时在线；更新 sessionToken 存在权限限制。
- **解决思路与方案**：实现了账号异地登录互斥（踢下线功能），复用云函数绕过前端安全规则更新 session。加入了紧急兜底密码并移除万能密码。

### 📅 2026-04-24：基础交互重构与加载优化

#### 1. 弹窗导航栏遮挡与交互不统一
- **问题现象与原因**：新建操作弹窗时，底部 TabBar 突兀地显示在遮罩上方；新建工地的交互与其他模块不一致。
- **解决思路与方案**：引入微信原生 `<root-portal>` 包裹所有全屏弹窗（新建待办、高级筛选、新建线索等），实现完美的沉浸式遮罩；将新建工地重构为底部浮窗组件，与新建待办等保持统一。
- **产品价值**：拉齐了全系统的新建操作交互体验，达到了原生 App 级别的使用质感。

#### 2. 待办排序混乱与跟进记录编辑失效
- **问题现象与原因**：待办列表最新数据沉在底部；编辑跟进记录时原内容不回显且保存失败。
- **解决思路与方案**：在数据获取时强制注入 `orderBy('createdAt', 'desc')` 排序；修复了回显绑定的 `data-id` 丢失问题，保存时记录 `editedAt` 时间戳并展示“已编辑”标签。

#### 3. 图片加载极慢与分享卡片降级
- **问题现象与原因**：列表页图片加载卡顿；普通工序分享时只显示公司大 LOGO 而非详情。
- **解决思路与方案**：
  - 上传端强制落实 `wx.compressImage`，确保所有图片在上传前被压缩。
  - 渲染端利用腾讯云 `imageMogr2/thumbnail` 强制输出 WebP 缩略图，极大缩短加载时间。
  - 修复 `generateShareImage` 的绘制时机，在工序验收完成后立即调用生成逻辑，确保工序分享卡片能正确渲染完工信息。

#### 4. 客户详情页设计排期按钮显示逻辑错误
- **问题现象与原因**：还未开启设计工作流时，客户详情页底部不应该显示“编辑排期”的按钮。
- **解决思路与方案**：在 `leadDetail/index.wxml` 中增加条件判断，只有当 `lead.designNodes` 存在且长度大于 0（即已开启设计工作流）时，才渲染底部的“编辑排期”按钮。

#### 5. 无需签字工序的验收提示优化
- **问题现象与原因**：不需要签字的普通工序，验收完还会错误地弹出“待客户确认”的提示。
- **解决思路与方案**：优化 `submitAcceptance` 成功后的提示语判断逻辑，仅当 `status` 为 `awaiting_signature` 时才提示“待客户确认”，否则直接提示“已验收通过”。

#### 6. 未读跟进记录红点提示缺失
- **问题现象与原因**：员工无法直观知道某个客户是否有新的未读跟进记录。
- **解决思路与方案**：引入未读消息检查机制，在客户列表和详情页“跟进记录”Tab 上增加红色小圆点提示，点击跟进记录 Tab 时自动标记为已读。

#### 7. 微信消息内容截断问题
- **问题现象与原因**：微信消息显示的内容（如工序验收）到一半就停止了，核心信息被前缀地址等占据。
- **解决思路与方案**：优化了工序验收的订阅消息文案结构，提取核心工序名称，避免被前缀地址等无用信息占据 20 字符限制导致截断。

#### 8. 客户签字后的反向通知缺失
- **问题现象与原因**：客户在分享页签完字提交后，员工端（项目经理、销售等）没有收到微信和系统的通知。
- **解决思路与方案**：在客户端签字提交成功的逻辑中，向该工地的相关负责人发送“客户已签字确认”的系统通知和微信订阅消息，并写入 `followUps` 系统记录。

### 💣 踩过的坑与阶段复盘 (V1.2 避坑指南)
在深水区重构中，我们触及了大量微信底层的 API 和复杂的组件交互，踩了不少“隐蔽的坑”：

- **坑 1：Mock 数据的“毒患”遗留**
  - **现象**：主材清单的图片在测试时一直“看起来挺好”，但一上线分享给客户，全变成了死图。排查才发现，原来一直用的是本地 mock 的临时路径，根本没调 `wx.cloud.uploadFile`。
  - **复盘**：前端 UI 跑通后，必须**第一时间拆除 Mock 代码**并接入真实 API。特别是涉及跨端分享的物料，绝不能抱有“本地能看就行”的侥幸心理。
- **坑 2：WXML 标签未闭合导致的“玄学白屏”**
  - **现象**：偶尔遇到整个页面白屏，控制台报出一个毫无关联的 `__route__ is not defined` 且伴随 `end tag missing`。
  - **复盘**：小程序的编译层极其脆弱。遇到这种“玄学报错”，第一时间不要去查 JS 逻辑，**先去查 WXML 里有没有少写 `</view>`** 或者组件层级嵌套死循环。
- **坑 3：订阅消息的“测试版黑洞”**
  - **现象**：在微信开发者工具里推送成功，一到手机体验版就收不到通知，且云函数也不报错。
  - **复盘**：微信官方的坑。云函数调用 `subscribeMessage.send` 默认只发给线上正式版。必须显式传入 `miniprogramState: 'trial'`。做微信生态，**环境参数的区分必须刻在骨子里**。
- **坑 4：排期算法的“理所当然”**
  - **现象**：一开始想当然地按工作日算排期（剔除周末），或者用固定的 `duration` 算天数，结果发现真实工地根本不按这套来，周末照样干活。
  - **复盘**：做产业互联网（如装修），千万不能用纯互联网打工人的思维去套。**必须深入业务现场**（自然日计算、并行施工、按天顺延），代码逻辑要为真实业务场景让步。

## 📱 V1.1.0 小程序突围期（中期：2026-04-23 及更早）
> **阶段思考**：系统推向一线时遭遇阻力，工长在工地不方便用电脑，客户查工地缺乏微信强提醒。战略重心全面向微信原生小程序倾斜，主攻“最后一公里”的现场体验。

#### 1. 多角色协同与财务权限体系建设 (核心特性)
- **业务痛点**：早期的系统仅支持单一项目经理和销售，无法应对复杂的团队协作；缺乏专门的财务核算视角。
- **解决思路与方案**：
  - **多分配机制**：重构了 `leads` 集合的人员字段，支持为同一个工地分配多个设计师、销售和项目经理。
  - **财务角色 (Finance)**：新增了只读性质的财务角色。针对未签单客户，财务看到的电话和地址等敏感信息全站打码并只读；客户签单后自动解密。
  - **人员搜索与展示**：员工列表页新增账号搜索功能；工地详情页的团队成员头像改为系统级的 `system avatar`，解决了旧版头像不统一的问题。
- **产品价值**：真正实现了公司各职能部门在同一套系统中的精细化协同与数据隔离。

#### 2. 大节点客户签字与流转架构重构 (核心特性)
- **业务痛点**：旧版甘特图排期只能线性执行（A做完做B），无法满足实际工地“交底后，水电泥木油多个小工序并行施工，最后统一验收”的真实业务场景。
- **解决思路与方案**：
  - **首尾特殊化模型**：创新性地设计了“首尾特殊化工序”架构。每个大节点的第一个小工序（如交底）和最后一个（如验收）设为签字节点。
  - **并行激活**：首节点签字完成后，中间的所有小工序同时被激活（并行状态）；直到尾节点验收签字，才流转到下一个大节点。
  - **动态排期引擎**：深度改造了 `recalculateGantt` 函数，工序完成后，后续所有工序的开始和结束时间会基于实际完成日期**动态顺延或提前**。
- **产品价值**：这是整个工程进度管理的核心引擎，使其排期逻辑完全贴合了真实装修工地的运作规律。

#### 3. 分享授权与隐私保护闭环 (核心特性)
- **业务痛点**：此前的工地进度分享链接只要拿到就能看，存在极大的客户隐私泄露风险；同时，员工在手机端测试客户视角时，总是被系统的 `isEmployee` 拦截。
- **解决思路与方案**：
  - **访客审核机制**：设计了全新的 `projectShare` 鉴权流。任何人首次打开分享链接均需输入手机号并提交查看申请，生成 `shareAccess` 待办记录。
  - **业主手机号静默授权**：如果业主本人的微信手机号与系统中录入的手机号匹配，则自动免密通过审核，极大降低业主的操作门槛。
  - **多身份视角解耦**：取消了前端强行使用 `isEmployee` 拦截页面展示的粗暴做法，改为根据实际分配记录判断权限，方便员工亲自测试客户视角。
- **产品价值**：在保证系统数据和客户隐私绝对安全的前提下，兼顾了极致的业主体验和内部测试便利性。

#### 4. 待办系统(Todo)业务闭环
- **业务痛点**：旧版待办只有新建和删除，无法追踪完成时间，也无法“重新打开”已误点的待办。
- **解决思路与方案**：
  - 增强待办表单，引入 `complete` 和 `reopen` 操作。
  - 在数据库层面精准记录 `completedAt` 时间戳。
  - 补充了通过 `todoForm` 发送给 assignee（指派人）时的 `senderName` 提取逻辑，解决通知中名字显示为 `[Object]` 的问题。

#### 5. 登录页体验与安全重构
- **业务痛点**：密码输入时无法确认是否输错；小程序本地缓存与服务端数据不一致，导致员工个人中心展示旧数据。
- **解决思路与方案**：
  - 登录页密码输入框增加了显示/隐藏明文的“小眼睛”图标，并修复了点击穿透无效的问题。
  - 重构登录生命周期：`doLoginSuccess` 回调中强制从 `users` 集合拉取最新员工数据覆盖本地缓存，确保任何在 PC 端的修改（如入职时间）在重新登录后立即生效。

#### 6. 遗留系统级 Bug 梳理 (Web端与底层数据)
- **问题现象与原因**：合同模块仍为假数据；报价单未关联 `leadId`；线索转项目未自动关联；密码明文存储存在安全漏洞。
- **解决思路与方案**：已在 `ISSUES_AND_RISKS.md` 中进行全面归档，列为 1.0 上线前/后的必须修复项（采用 bcrypt 哈希、补充 `tcbQuery` 导入等）。

### 💣 踩过的坑与阶段复盘 (V1.1 避坑指南)
在这个从小程序“开荒”并快速落地的阶段，由于对云开发的机制不够熟悉，我们碰到了大量权限、缓存与角色隔离的坑：

- **坑 1：分享页面的“员工特权”拦截测试**
  - **现象**：为了防止客户看到内部操作按钮，我们在页面顶部加上了 `if (isEmployee)` 的拦截。结果自己用员工号测试客户分享链接时，功能全没了，根本测不了客户视角。
  - **复盘**：角色权限判断千万不能做这种“一刀切”的 UI 隐藏。应该解耦**“身份”**与**“场景”**，比如通过具体的 `shareAccess` 记录权限，而不是简单依赖当前登录账号的角色。
- **坑 2：云数据库“更新了但没完全更新”的权限陷阱**
  - **现象**：小程序端调用 `db.collection('users').update()` 刷新 `sessionToken`，代码执行成功不报错，但返回的却是 `updated: 0`。
  - **复盘**：这是云开发最容易忽视的安全规则坑。非创建者去修改管理员创建的记录，默认会被静默拦截。遇到此类越权更新操作，**必须走云函数**（如 `bindOpenId`）绕过前端限制，决不能图省事在客户端直接 `db.update`。
- **坑 3：生命周期与缓存的脱节**
  - **现象**：在个人中心页，发现明明 PC 网页端刚改了员工入职时间，手机端重登录还是旧数据。
  - **复盘**：小程序的“离线优先”和本地缓存机制是一把双刃剑。涉及个人资料等强一致性要求的数据，必须在 `doLoginSuccess` 或 `onShow` 时**强制拉取云端覆盖本地**，绝不能盲信 `wx.getStorageSync`。

## 💻 V1.0.0 Web 端基建期（早期：从 PRD 启动）
> **阶段思考**：在项目启动之初，为了解决 20 人装修团队极度依赖 Excel、数据不互通的管理乱象，我们决定先搭建一个重度管理的“PC 大本营”。
> **技术选型**：采用 `Next.js 14 (App Router)` + `Tailwind CSS` 构建全栈 Web 应用，后端直接对接微信云开发（TCB）的 Node.js SDK，实现了无服务器（Serverless）架构的快速起步。

### ✨ 核心基建与业务模型落地
- **线索状态机与 A/B/C/D 意向评级系统**：强制销售在录入时进行客户评级，摒弃大锅饭，精准锁定高意向客户。
- **CRM 到 ERP 的单向流转模型**：实现了“客户(Lead) -> 报价(Quote) -> 签单建群 -> 工地项目(Project)”的单向业务主线。
- **极简的 8 大标准施工节点流转模型**：坚决摒弃市面上 SaaS 繁琐的子工序，敲定极简的 8 大节点（交底、水电、泥瓦、木工等），打好了后来被小程序复用的核心数据结构。
- **多角色权限体系搭建**：在云数据库中构建了 admin（老板）、sales（销售）、designer（设计师）、manager（工长）等基础角色权限矩阵。

### 💣 踩过的坑与技术债反思 (坑与教训)
在这个阶段，由于初期追求业务逻辑快速上线，我们在 Web 端留下了不少“坑”和技术债，这些在后期复盘中都成了宝贵的经验：

- **坑 1：数据模型与实际存储的“两层皮”**
  - **现象**：项目中定义了传统的 Mongoose Schema（如 `web/src/models/Project.ts`），但在实际调用 TCB 云数据库时，写入的字段完全是另一套（如 `nodesData`, `startDate` 等）。
  - **反思**：Serverless 无模式（Schemaless）数据库极其灵活，但也极易导致字段失控。最终我们废弃了无用的 Model 文件，全面拥抱 TypeScript 接口定义来约束云数据库操作。
- **坑 2：关联数据“断链”与数据孤岛**
  - **现象**：Web 端的报价单（Quotes）只存了自身数据，没有存 `leadId` 或 `projectId`；线索转为“已签单”后，系统并没有自动创建关联的 Project 记录，必须人工再去建一个项目。
  - **反思**：初期过度解耦导致了数据断层。在 CRM/ERP 系统中，核心单据（报价单、合同、项目）的生成必须强绑定源头单据（线索 ID）。
- **坑 3：并发冲突与“硬编码”陷阱**
  - **现象**：生成客户编号采用的是“查当前最大编号 + 1”，导致多销售同时录入时出现编号重复；系统查询接口全部硬编码了 `.limit(100)` 导致数据超过 100 条后直接“消失”；甚至 Web 端合同详情页（`contracts/[id]/page.tsx`）的状态是用 `id % 3` 这种数学运算模拟的假数据。
  - **反思**：业务增长比想象中快。自增 ID 必须用原子操作或时间戳+随机数；分页功能必须从 Day 1 就设计好，不能用硬编码 limit 糊弄。
- **坑 4：安全意识的缺失（明文与硬编码）**
  - **现象**：Web 端创建员工时，密码竟然以 `passwordPlain` 明文存入数据库；而且微信的 `APPSECRET` 直接写死在了 `route.ts` 接口代码里。
  - **反思**：这是最惨痛的教训。后来我们制定了严格的安全底线规范：所有敏感密钥必须移入 `.env.local` 环境变量，密码后续必须引入 bcrypt 进行哈希加密改造。
- **坑 5：通知系统的“静态化”**
  - **现象**：Web 端的通知列表（Notifications）做好了，但全是空的，因为没有任何一个业务动作（如分配线索、节点完成）去触发它。
  - **反思**：没有“自动触发器”的通知系统就是摆设，这也直接促使了我们在后期开发小程序时，把“系统跟进记录 + 微信订阅消息”强制与业务动作绑死。

## 🌟 全局产品演进复盘与核心方法论

回顾从 V1.0 的 Web 基建，到 V1.1 的小程序突围，再到如今 V1.2 的深水区重构，我们不仅是在写代码、修 Bug，更是在不断迭代**“对装修产业数字化的认知”**。以下是我们总结出的三条全局核心方法论：

### 1. 架构演进：从“大而全的管控”走向“小而美的协同”
在 V1.0 时期，我们用 Web 端（Next.js）搭建了庞大的底座，试图用传统的 CRM+ERP 逻辑把所有人和数据圈在一起。但这忽略了装修行业的根本特性：**核心工作是在没有电脑的毛坯房现场发生的。**
后期的重心果断转向微信小程序（V1.1、V1.2），实现了从“管控思维”向“协同思维”的转变。工长可以拍照发进度，业主可以在微信直接签字，这打通了数字化落地的最后一公里。

### 2. 数据哲学：从“全站实时强一致”走向“场景解耦快照”
程序员的思维惯性是“数据变了，全端都要立刻变”，但这在 ToC 的业务中极其危险。
比如 V1.2 中的**主材清单**，如果员工改了价格，客户端跟着实时变，会导致信任危机。我们最终摸索出了**“Snapshot 快照解耦机制”**：员工操作的是“实时草稿”，点击发送时生成“版本快照”，客户看到的永远是被锁定的快照。**业务的安全性远大于技术上的“实时性”。**

### 3. 业务模型：不要教业务做事，要让代码适配现场
最大的教训来自于 V1.1 和 V1.2 阶段的**排期与甘特图**开发。最初我们想当然地把市面上 SaaS 那套“父子任务、工作日剔除、线性流转”直接搬过来，结果现场根本用不起来（因为工地是多工种并行、周末无休的）。
最终我们重构了**“首尾特殊化工序、并行施工、自然日计算”**。这让我们深刻意识到：产业互联网的本质是**贴地飞行**。只有深入业务现场，才能写出有生命力的系统，而不是一个精致但无用的摆设。
