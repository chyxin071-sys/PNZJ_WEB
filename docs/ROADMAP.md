### Step 1 — 修复员工改名功能失效的 Bug

**问题**：管理员在网页端修改员工姓名时，关联数据（线索、项目、待办等）不会同步更新，但页面不会报错，你根本不知道出问题了。

**要改的文件**：`web/src/app/api/employees/[id]/route.ts`

**改什么**：第 2 行的 import 里补上 `tcbQuery`

```
改前：import { tcbUpdate, tcbDelete } from '@/lib/wechat-tcb';
改后：import { tcbQuery, tcbUpdate, tcbDelete } from '@/lib/wechat-tcb';
```

**验收方法**：
1. 管理员登录网页端，进入员工管理
2. 修改某个员工的姓名（比如"张三"改为"张三三"）
3. 去线索列表，找到该员工负责的线索，确认销售/设计师字段已同步更新
4. 去项目列表，确认项目经理字段已同步更新

---

### Step 2 — 修复个人主页待办数量显示为 0 的 Bug

**问题**：个人主页的"我的待办"数量永远显示 0，因为字段名写错了。

**要改的文件**：`web/src/app/profile/page.tsx`

**改什么**：第 69 行，把 `t.assignedTo?.name === user.name` 改为正确的 assignees 数组判断

```
改前：todos.filter((t: any) => t.status === 'pending' && (t.assignedTo?.name === user.name || t.createdBy?.name === user.name))
改后：todos.filter((t: any) => t.status === 'pending' && (t.assignees?.some((a: any) => a.name === user.name) || t.creatorName === user.name))
```

**验收方法**：
1. 登录一个有待办任务的账号
2. 进入个人主页，确认"我的待办"数量不为 0，且与实际待办数量一致

### Step 3 — 把 AppSecret 移出代码

**问题**：微信的 APPID 和 APPSECRET 直接写在源代码里，任何能看到代码的人都能拿到。

**要改的文件**：
- `web/src/lib/wechat-tcb.ts`
- `web/src/app/api/auth/login/route.ts`
- `web/src/app/api/upload/route.ts`

**改什么**：
1. 在 `web/` 目录下创建 `.env.local` 文件（这个文件不会被 git 提交）
2. 把敏感信息写进去：
```
WECHAT_APPID=wxc9f24e9a9f57bc7a
WECHAT_APPSECRET=41bc50ad8d277918037ea0107d0a07b0
NEXT_PUBLIC_TCB_ENV_ID=cloud1-8grodf5s3006f004
JWT_SECRET=pnzj-super-secret-key-123
```
3. 代码里改为读取环境变量：`process.env.WECHAT_APPSECRET`

**验收方法**：
1. 确认 `.env.local` 文件存在且不在 git 追踪里（`git status` 看不到它）
2. 重启 Next.js 开发服务器，登录功能正常
3. 上传文件功能正常

### Step 5 — 报价单关联线索

**问题**：报价单和客户线索是两个孤立的数据，无法互相跳转。

**要改的内容**：
1. 报价单数据结构增加 `leadId`、`leadName` 字段
2. 新建报价单时，必须先选择关联的客户（从线索列表里选）
3. 线索详情页的"报价单"入口，改为查询 `leadId = 当前线索ID` 的报价单
4. 报价单列表增加"客户名"列，可点击跳转到线索详情

**涉及文件**：
- `web/src/app/quotes/new/page.tsx`（新建报价单页面）
- `web/src/app/api/quotes/route.ts`（GET 接口增加 leadId 筛选）
- `web/src/app/leads/[id]/page.tsx`（线索详情页报价单入口）
- `mini-program/pages/quoteDetail/index.js`

**验收方法**：
1. 新建报价单时，必须选择一个客户，保存后 `leadId` 字段有值
2. 进入某个线索详情，点击"报价单"，能看到该客户的所有报价单
3. 报价单列表里能看到客户名，点击能跳转到线索详情

### Step 6 — 合同模块接入真实数据

**问题**：合同管理页面全是假数据，无法实际使用。

**要改的内容**：
1. 在 `leads` 集合里增加合同相关字段：
   - `contractStatus`：合同状态（待签/已签/执行中/已结项）
   - `contractSignDate`：签约日期
   - `contractAmount`：合同金额
   - `payments`：付款节点数组（每项含阶段名、金额、状态、收款日期）
2. 合同详情页改为从数据库读取真实数据
3. 合同状态改为真实字段，不再用 `id % 3` 模拟

**涉及文件**：
- `web/src/app/contracts/[id]/page.tsx`（合同详情页）
- `web/src/app/contracts/page.tsx`（合同列表页）
- `web/src/app/api/leads/[id]/route.ts`（PUT 接口支持更新合同字段）

**验收方法**：
1. 进入一个"已签单"状态的线索，点击"合同"，能看到真实的客户信息
2. 确认付款节点可以手动标记为"已收"，刷新后状态保留
3. 合同列表的状态标签与数据库里的 `contractStatus` 字段一致

### Step 7 — 网页端新建工地对齐小程序逻辑

**决策**：签单不自动创建工地草稿，由项目经理手动新建，过一遍手更清晰。两端流程统一。

**问题**：网页端新建工地时没有"选关联客户"的功能，项目经理需要手动填所有信息；线索详情页"查看工地"按钮直接跳工地列表首页，没有带 `leadId` 筛选，找不到对应工地。

**要改的内容**：
1. 网页端新建工地页面增加"选关联客户"入口（从已签单线索里选），选完自动填入客户名、地址、面积、销售、设计师
2. 线索详情页"查看工地"按钮，改为先查有没有关联工地：有则跳工地详情，没有则跳新建工地页面（带 `leadId` 参数）
3. 工地列表页支持 `leadId` URL 参数筛选

**涉及文件**：
- `web/src/app/projects/page.tsx`（新建工地入口 + leadId 筛选）
- `web/src/app/leads/[id]/page.tsx`（查看工地按钮逻辑）
- `web/src/app/api/projects/route.ts`（GET 接口支持 leadId 筛选）

**验收方法**：
1. 进入一个已签单线索，点击"查看工地"，如果没有工地则跳到新建页面且客户信息已自动填入
2. 新建工地保存后，回到线索详情点"查看工地"，能直接跳到该工地详情
3. 工地列表带 `?leadId=xxx` 参数时，只显示该客户的工地

### Step 7b — 跟进记录时间格式统一

**问题**：小程序写入的 `createdAt` 是字符串格式（`"2025-01-01 10:00"`），网页端写入的是对象格式（`{ $date: timestamp }`），两端互相读取时时间显示可能出错。

**要改的内容**：统一改为网页端格式（`{ $date: timestamp }`），小程序写入时也改用 `db.serverDate()`

**涉及文件**：
- `mini-program/pages/addFollowUp/index.js`
- `mini-program/pages/leadDetail/index.js`（`addSystemFollowUp` 函数）

### Step 10 — 关键业务事件自动触发通知

**问题**：通知只能手动发，业务事件发生时没有自动提醒。

**要实现的自动通知**：

| 触发事件 | 通知接收人 | 通知内容示例 |
|---------|-----------|------------|
| 线索分配给销售 | 被分配的销售 | "你有一条新线索：李四（138****0000）" |
| 线索分配给设计师 | 被分配的设计师 | "你有一条新设计任务：李四的方案" |
| 项目节点完成 | admin + 项目相关人员 | "张三的工地【水电】节点已验收完成" |
| 项目发生延期 | admin + 项目经理 | "张三的工地【瓦工】节点延期 3 天" |
| 待办到期提醒 | 待办负责人 | "你有一个待办即将到期：XXX" |
| 新建报价单 | admin | "销售小王为客户李四创建了报价单" |

**涉及文件**：
- `web/src/app/api/leads/[id]/route.ts`（分配人员时触发）
- `web/src/app/api/projects/[id]/route.ts`（节点完成/延期时触发）
- `web/src/app/api/todos/route.ts`（创建待办时触发）
- `web/src/app/api/quotes/route.ts`（新建报价单时触发）

**验收方法**：
1. 把一条线索分配给某个销售，该销售登录后能在通知中心看到提醒
2. 项目节点验收完成后，admin 能收到通知
3. 项目发生延期后，admin 和项目经理能收到通知

### Step 11 — 数据查询分页

**问题**：所有列表最多查 100 条，业务增长后数据会"消失"。

**要改的内容**：
1. 所有 GET 接口支持 `page` 和 `pageSize` 参数
2. 前端列表页增加分页控件（上一页/下一页/跳页）

**涉及文件**：所有 `web/src/app/api/*/route.ts` 的 GET 接口

**验收方法**：
1. 数据库里有超过 100 条线索时，第 2 页能正常加载
2. 分页控件显示正确的总数和当前页码

---

### Step 12 — 客户编号并发冲突修复

**问题**：同时录入两条线索时，可能生成相同的客户编号。

**要改的内容**：
- 客户编号生成逻辑改为：`P${year}${时间戳后6位}`

**验收方法**：
1. 快速连续新建 3 条线索
2. 确认 3 条线索的客户编号各不相同

### Step 13 — 材料大厅与库存管理拆分

**问题**：材料大厅（产品目录）和库存数量混在一张表里，概念混淆，无法记录出入库流水。

**要改的内容**：
1. `materials` 集合只保留产品信息（名称、规格、单价、单位、分类、供应商）
2. 新建 `inventory` 集合记录出入库流水（关联产品、类型、数量、关联工地、操作人、时间）
3. 新建库存管理页面，支持入库/出库操作，实时显示各产品当前库存

**涉及文件**：
- `web/src/app/materials/page.tsx`
- `web/src/app/inventory/page.tsx`（新建）
- `web/src/app/api/inventory/route.ts`（新建）

## 阶段五：财务管理（v2.0.0）
> 目标：建立完整的财务视图，覆盖收款、成本、利润
> 预计工作量：大，需要仔细梳理业务流程后再开发
> 状态：待规划，暂不开发

**初步思路（待细化）**：

- 收款管理：基于合同的分期收款（已在 Step 6 中打基础）
- 工地成本：每个工地的材料用量、人工费、零散支出记录
- 店面支出：日常运营费用流水（水电、房租、工资等）
- 利润报表：签单金额 - 工地成本 - 店面分摊 = 毛利润

> 注意：报价单是给客户看的对外文件，工地成本是内部核算，两者数据来源不同，不要混用。
